문서 개요

이 문서는 SPERO의 다양한 기능과 UI/UX의 운용을 원활하게 수행하기 위해서 작성된 사용자 설명서로 SPERO의 기본적인 개요부터 사용자 운용을 위한 인터페이스, 메뉴 사용 방법, 선택사항, 기능 수행 절차와 같은 SPERO를 실행하기 위해 필요한 정보에 대해서 설명한다.

SPERO의 구조

SPERO는 IaaS, PaaS, Service를 통합하여 관리하고, SPERO사용자는 시간이나 장소에 제한을 받지 않고, 언제 어디서나 사용할 수 있으며, SSO를 통해 권한이 부여된 SPERO사용자의 권한을 다른 Application에서 동시에 사용할 수 있다. 또한, 정책 기반의 자동 확장성을 통해 약속된 성능이 보장된다.

SPERO에 탑재되는 모든 서비스의 운영, 유지 보수 및 가용성 관리는 SPERO에서 지원한다. SPERO는 자동화된 확인 작업을 통해 배포 후, 새 구성 요소와 기능을 사용할 수 있다. 즉, SPERO운영자는 항상 최신 SPERO응용 Program을 사용할 수 있으며, SPERO의 새로운 서비스는 Cluster 또는 Tenant 단위로 관리된다.

SPERO는 고객이 Single Package로 요구하는 다양한 솔루션과 정책을 제공하는 Unified UI/UX Solution이다. SPERO는 전문 그래픽 디자이너가 설계한 UI/UX를 제공하여 보다 편안하고, 작동하기 쉬운 화면을 제공하며, 하나의 UI/UX를 통해 다양한 Cloud Service를 통합하여 관리함으로써 운영의 편리성, 직관적인 UI/UX 및 서비스 기반 운영을 제공한다.

SPERO는 통합 UI/UX를 제공하는 것 외에도 고객이 원하는 메뉴를 수행할 수 있는 Workflow 기반 운영 서비스를 제공함으로써 복잡한 작업을 단계별로 연결하는 One-stop Service를 제공하여 작업을 쉽게 수행할 수 있도록 지원한다.

SPERO는 작업하는 동안 수행된 작업 내역을 기록하고, 관리하며, 작업이 완료 또는 종료되지 않은 경우, 작업자가 일정 시간이 지난 후, 다음 작업을 계속 수행할 수 있는 기능과 작업 흐름을 이어서 수행하는 환경을 제공한다. SPERO는 작업의 진행 상황과 완료를 통해 진행 상황을 관리하므로 현재 작업의 진행 상황과 완료 상태를 시각적으로 확인할 수 있다.

SPERO는 IaaS, PaaS 및 Application Service를 통합하여 서비스를 관리하고, 감시하며, 조회하는 통합 관리 기능을 제공하여 Cloud 사용자에게 안전하고 편리한 Infrastructure, Platform 및 Application 관리기능을 제공한다.

특징

SPERO구성의 특징은 다음과 같다.

Frontend(FE)와 Backend(BE)로 구성한다.

Frontend와 Backend 구간의 모든 통신은 AGF(API gateway Framework)를 경유해야만 사용할 수 있다.

API Gateway 하위의 API 호출에는 내부 URL로 변환하여 전달한다.

SPERO는 다른 Cloud Services를 쉽게 수용 또는 운용할 수 있도록 Plug-in하여 구성할 수 있으며, 다양한 API를 추가하여 연동할 수 있다.

SPERO는 한 번의 로그인으로 API Gateway 하위의 모든 Solutions (or Tools)과 통신이 가능하다.

그림 2-1 SPERO구성 간 작업 흐름도

Frontend (FE)

SPERO의 Frontend는 ‘Vue.js3’을 이용하여 화면 페이지를 구현하였으며, 이 화면 페이지는 Frontend만을 위한 Web Server(Backend for Frontend, 이하 ‘BFF’로 부른다.)를 제공한다. BFF는 “Spring Boot”로 구현되어 있다.

BFF와 SPERO Server 간에는 REST API로 통신한다. 모든 메시지는 Server의 API gateway인 “Kong”으로 전달된다. 이때, JWT token을 이용하여 인증 여부를 확인한다.

Tree Model 및 Event Polling 결과는 SPERO서버에 일정 주기마다 REST API를 호출하여 데이터를 수집한다. Tree 형상 변경 시, 실시간 반영이 아닌 Tree Viewer 동기화 기능을 이용하거나 다음 Polling time에 가져온 정보를 기반으로 동기화가 실행된다.

Backend (BE)

SPERO Backend(BE) 서비스는 SPERO MNO (Management and Orchestration)을 위한 서비스 모듈을 제공한다. SPERO의 BE를 지원하기 위해 사용하는 도구는 AGF(API gateway Framework), AMF(Authentication Management Framework), CMF(Configuration Management Framework), UMF(User Management Framework), LMF(Log Management Framework), AIF(Active Inventory Framework), SO(Service/App Orchestrator), MVC(Multi-VIM/Cloud), ACF(Application Controller Framework), DCM(Data Collector & Mediator) 등이 있다.

AGF (API gateway Framework)

SPERO에서 제공되는 모든 서비스는 REST API로 통신하며, 서비스를 사용하기 위해서 SPERO에 로그인을 해야 하고, 인증과 인가를 취득해야 사용할 수 있다.

Frontend에서 요청하는 모든 메시지는 AMF(API Gateway)를 통해서 이용된다.

Kong 오픈소스를 기반으로 구성되었다.

AMF( Authentication Management Framework)

Keycloak과 연동하여 사용자 인증 서비스를 제공한다.

인증 방식은 OIDC를 사용한다.

CMF (Configuration Management Framework)

IaaS/PaaS에서 주기적으로 형상 정보를 수집(Polling)하여 형상 테이블에 저장한다.

저장된 형상 테이블은 Tree 형태로 볼 수 있도록 재구성한다.

이벤트 정보는 SPERO Server/IaaS/PaaS에서 주기적으로 수집하여 장애 테이블에 저장한다.

UMF (User Management Framework)

SPERO에서 제공하는 인증 서비스를 제외한 대부분의 서비스는 UMF (User Management Framework)에서 제공한다.

인증 서비스를 사용하기 위해서는 SPERO계정으로 로그인을 해야 한다.

SPERO에서 조회하고 저장하는 데이터는 MariaDB에 저장된다.

이벤트 관리 등을 수행한다.

LMF (Log Management Framework)

로그 데이터를 수집할 수 있는 기능을 제공한다.

수집한 로그 데이터는 DB에 저장된다.

수집하고 있는 로그 데이터는 API Gateway에서 수행한 명령어 이력 데이터와 시스템 운용 이력 데이터이다.

AIF (Active Inventory Framework)

형상 정보, 장애 정보, 로그 정보는 모두 MariaDB의 DBMS에 저장한다.

빠른 조회를 위하여 Redis DB와 연동하여 사용한다.

SO (Service/App Orchestrator)

LCM을 workflow 기반으로 관리하며, 오케스트레이션 관련 내부 서브 시스템 (e.g., ACF, MVC) 등과 연동한다.

서비스 생성/삭제/중단/재개 등의 동작을 수행한다.

Airflow 오픈소스 기반으로 구성되어 있다.

MVC (Multi-VIM/Cloud)

Infrastructure Adaptation 기능을 담당한다.

infrastructure (private cloud 또는 public cloud)와 연동하는 interface를 관리하는 기능을 제공한다.

Terraform 어플리케이션 기반으로 구성되어 있다.

ACF (Application Controller Framework)

application의 설치/삭제/시작/중단 및 configuration 갱신 등의 기능을 담당한다.

모니터링 agent의 설치 및 수집 정보 변경 기능도 담당한다.

API 기반의 온프레미스 모니터링 기능을 제공한다.

Ansible 오픈소스 기반으로 구성되어 있다.

DCM (Data Collector & Mediator)

SPERO에서 agent로부터 수집되는 메트릭을 저장 관리한다.

TSDB인 influxDB기반으로 구성되어 있다.

API Gateway - Kong 연동

API Gateway는 SPERO로 유입되는 트래픽에 대한 관리 및 제어를 수행하고, SPERO의 고유 기능을 수행하는 Backend의 모든 API 서버와 연동한다. API Gateway는 다음과 같은 특징을 갖는다.

API Gateway에는 Kong을 도입하여 사용한다.

모든 API는 API Gateway를 거쳐서 호출한다.

API Gateway에서의 URI 변환은 아래 표의 규칙에 따라 수행한다.

DBMS

SPERO운용에 필요한 각종 데이터의 저장, 검색 등의 관리를 위한 데이터베이스 시스템을 사용한다. RDBMS인 MariaDB를 사용하고 성능 향상을 도모하기 위해 Redis를 추가로 적용하였다.

시스템 운용에 필요한 데이터의 관리를 위해 데이터베이스 시스템을 구성한다.

기본 데이터베이스 시스템으로는 RDBMS인 MariaDB를 구성한다.

빈번하게 제공되어야 하는 데이터에 대한 향상된 성능 제공을 위해 In-Memory Storage 방식인 Redis를 추가 구성한다.

데이터베이스는 SPERO통합 관리 시스템을 지원하며 다양한 데이터/정보의 저장 및 관리에 사용된다.

SPERO는 IaaS, PaaS, Service에 대한 통합 관리를 수행하기 위한 서비스이다. SPERO를 사용하기 위해 로그인 후, SPERO가 수용하고 있는 솔루션들과 연동하기 위해서 통합 계정 관리를 수행하는 Keycloak과 연동한다. 또한, SPERO제품의 외부 보안 및 보안 인증을 위하여 AGF (API gateway)인 Kong 과 연동한다.

로그인/아웃

SPERO를 운용하기 위해서 SPERO Management 시스템으로 로그인해야 접속이 가능하다. SPERO는 Manager 시스템으로만 로그인하여 SPERO서비스 및 연동되는 제품의 IaaS, PaaS, Service의 자원들로 접근이 가능하다. 단, SPERO의 사용자 계정은 사전에 등록하여야 한다.

SPERO의 로그인 특징은 다음과 같다.

사용자가 ID/PW를 입력하여 SPERO Backend로 로그인한다.

사용자 최초 접속 시, 반드시 비밀번호를 변경한다.

이미 인증되어 사용 중인 계정으로 중복 로그인 요청이 들어오면 중복 접속 요청되었음을 사용자에게 알리고, 이전 접속 정보는 삭제한다.

사용자가 ID/PW 입력에서 허용 횟수(기본 5회) 이상 잘못 시도할 경우, 더 이상의 접속 시도는 불가하도록 계정은 자동으로 차단되고, 다시 관리자의 승인을 받아야만 사용이 가능하다.

사용자의 요청에 의하여 로그아웃이 가능하다.

SPERO에 접속하기 위해 크롬을 실행한다.

다국어 (한국어)를 선택한다. 기본으로 한국어가 선택되며, 다른 언어를 선택하여 실행하면 화면의 내용이 선택한 언어로 변환된다.

ID 저장을 체크한다.

계정 등록 후, 최초 로그인 시, 비밀번호 변경 페이지로 이동한다.

Tree Viewer 영역

SPERO Backend는 On-Premise, IaaS, PaaS, Service 정보를 주기적으로 조회하여 Cloud, Cluster, Node로 구성된 계층구조(Tree) 형태의 형상 정보를 생성한다. 또한, IaaS의 형상 정보를 수집 (인스턴스 정보)을 담당하고, PaaS의 형상 정보(Cloud(Platform), Cluster, Node, Pod)를 자동으로 수집하기 위해 사용한다.

SPERO Frontend는 주기적으로 데이터를 조회하여 최신 상태로 유지한다. 또한, 사용자에게는 권한이 부여된 Cluster의 범위 내에서만 형상 정보가 제공된다. Frontend에서 제공하는 Tree 정보는 다음의 특징이 있다.

Tree 정보는 PaaS, IaaS및 Service 로 구분되며, PaaS는 Cloud, Cluster, Node의 4단계의 계층 구조(hierarchy)를 갖는다. IaaS는 CSP, 클라우드, 인스턴스를 포함하는 3단계의 계층 구조를 가지며, PaaS의 노드와 Mapping되지 않는 인스턴스 목록을 표시한다.

권한이 있는 Cluster만 형상 정보에 포함한다.

형상 정보는 주기적인 정보 수집으로 자동 동기화한다.

시스템의 상태 정보를 함께 수집한다.

그룹 관리 기능은 사용자가 소속된 그룹 정보를 관리하는 화면이다. 그룹을 생성하여 그룹에 권한을 부여하며, 그룹에 속한 사용자들에게 동일한 권한을 부여하는 기능을 제공한다.

관리자 메뉴의 그룹 관리를 선택하여 그룹 조회, 생성, 수정, 삭제할 수 있다.

그룹 리스트에서 그룹 이름을 수정할 한 개의 그룹 명을 선택하고, 수정 () 아이콘을 누르면 그룹 수정 팝업이 표시된다. 그룹 명은 50자를 넘어서는 안된다.

1순위 - 사용자 별 부여된 권한

2순위 - 등급에 부여된 권한

3순위 - 그룹에 부여된 권한

그림 9-15 서비스-서비스 차트

이벤트/통계

이벤트/통계는 여러 클라우드 환경에서 발생하는 알람, 경고와 같은 이벤트와, 리소스에 대한 통계 데이터를 주기적으로 수집하고 이를 사용자가 SPERO를 통해 한곳에서 확인할 수 있는 기능을 제공한다.

이벤트

이벤트는 Pod, Cluster, VM 과 같은 각 클라우드의 인프라에서 발생하는 이벤트를 주기적으로 수집하고 이를 SPERO에서 정의된 형태로 변환하여 SPERO의 테이블에 적재한다.

이벤트의 종류, 상태 등을 기준으로 분류한다.

이벤트 종류를 바탕으로 장애 발생 또는 일반 알람을 구분하여 저장한다.

이벤트는 발생 일시에 따라 순차적으로 조회할 수 있도록 저장된다.

현재 발생 중인 알람은 별도의 API를 사용하여 일반 이벤트와 분리하여 제공한다.

SPERO이벤트 메뉴에서는 현재 알람 조회와 이벤트 이력을 조회할 수 있다.

그림 10-1 이벤트 조회 메뉴

현재 알람

현재 알람은 SPERO를 통해서 생성 또는 모니터링 되고 있는 인프라에서 현재 발생한 이벤트를 표시한다. 각 이벤트는 심각도와 함께 최신 순으로 표시된다.

현재 알람을 조회하는 메뉴는 이벤트/통계 메뉴를 선택한 후, 현재 알람을 통하여 조회할 수 있다.

그림 10-2 현재 알람

알람 리스트에 출력되는 항목은 다음과 같다.

표 10-1 알람 리스트 출력 항목

현재 발생한 알람 정보가 등급별로 총합으로 표시한다. 알람 등급은 Critical, Major, Minor, Warning으로 구분하여 표시된다.

그림 10-3 알람 Summary

특정 알람을 클릭한 다음에 Clear () 버튼을 클릭하면 알람이 해제된다.

상세 () 버튼을 클릭하여 알람 발생 기간, 알람 등급, 알람 명을 입력하여 지정된 필터링 조건에 맞는 알람만 검색할 수 있다.

그림 10-4 현재 알람 상세 검색

이벤트 이력

이벤트 이력은 발생한 이벤트(알람, 노티, 상태 메시지)를 표시한다. 이벤트 이력에 표시되는 알람은 발생된 모든 알람을 표시한다.

이벤트/통계 메뉴를 선택한 후, 이벤트 이력을 클릭하여 이벤트 이력을 조회할 수 있다.

이벤트 이력에서 출력되는 이벤트 이력 항목은 현재 알람과 동일하다.

그림 10-5  이벤트 이력 조회

통계

통계는 각 인프라에서 사용 중인 리소스 데이터를 수집하고 이를 라이프 사이클, 성능, 미터링으로 데이터를 통계 내어 분류하고 표시한다.

SPERO에서 정의된 기준에 따라 수집된 데이터를 라이프사이클, 성능, 미터링으로 데이터를 분류한다.

리소스의 생성 그리고 종류에 따른 데이터는 라이프사이클에 저장된다.

서비스/인스턴스에서 사용된 리소스를 일/주/월로 통계 내어 성능 데이터에 저장된다.

설정된 미터링 주기에 따라 수집된 리소스 사용량 데이터를 서비스/인스턴스로 분류하여 미터링 데이터에 저장된다.

SPERO이벤트 메뉴에서는 라이프사이클, 성능, 미터링 데이터를 조회할 수 있다.

로그 관리

서비스

서비스는 다중의 어플리케이션들, VM 인스턴스들, 서브넷, 모니터링 에이전트 등으로 구성된 논리적 집합이며 퍼블릿/프라이빗 클라우드 환경에서 특정 서비스를 제공하기 위한 집합체이다. 사용자는 CSP에 직접 접속하거나 유지 관리할 필요 없이 SPERO를 통해 서비스를 배포, 설치, 수정, 삭제, 정보 조회 등이 가능하다.

그림 12-1 서비스 메뉴

서비스 디자인

서비스 디자인 메뉴에서는 서비스에 대한 형상을 정의한 후 정의된 서비스 형상(서비스 템플릿)을 카탈로그에 저장하는 기능을 한다.

서비스 디자인은 서비스 메뉴를 선택 후, 서비스 디자인을 통해서 확인할 수 있다.

그림 12-2 서비스 디자인 화면

서비스 디자인 메뉴 또는 카탈로그 화면에서 “+onboard” 버튼을 클릭한다

General 탭에서 템플릿 명(필수 값), 설명, 그리고 템플릿 타입을 지정 후 저장()버튼을 클릭한다.

그림 12-3 General 탭 화면

Resource 탭의 인스턴스 목록에서 생성할 인스턴스의 정보(인스턴스 명, CPU, Memory, Extra-Disk, OS, Version)를 설정 후 추가() 버튼을 클릭한다. (Extra-Disk 값을 “0”으로 선택된 OS에서 요구하는 최소 disk 만큼의 용량이 설정된다.)

그림 12-4 Resource 탭 인스턴스 목록

Resource 탭의 서브넷 목록에서 서브넷 명과 public IP 사용유무 체크 후 추가() 버튼을 클릭하고, 저장() 버튼을 클릭한다. (만약 public IP 사용유무에서 “N” 선택 시 해당 서브넷이 할당된 인스턴스는 private IP 만 할당 받게 된다.)

그림 12-5 Resource 탭 서브넷 목록

Application 탭에서 추가할 어플리케이션이 존재하면, 어플리케이션 정보(어플리케이션 유형, 어플리케이션 명, 포트, URI)를 입력 후 추가() 버튼을 클릭하고, 존재하지 않다면 저장() 버튼을 클릭한다.

그림 12-6 Application 탭 화면

Application 디자인 시 필수적으로 관련 설치 파일을 추가해줘야 하며, () 버튼에서 “Property 수정”을 선택/클릭 후 생성되는 “Property” 팝업에서 “create” 타입에 설치 파일을 등록하여 한다.

그림 12-7 Application 탭 Property 수정 화면

() 버튼을 클릭하여 파일 입력 창에서 파일 선택 부분을 클릭하여 script file을 불러온 후 적용() 버튼을 클릭한다.

그림 12-8 Application 탭 파일 입력 화면

Application의 설정이 완료되면, 저장() 버튼을 클릭한다.

Agent 설정 화면에서 수집할Application-specific metric의 유무에 따라 파일 플러그인 사용 여부에서 “Y” 또는 “N”을 선택한다. “N” 선택 시 추가()버튼을 클릭 후 저장() 버튼을 클릭한다.

그림 12-9 Agent 탭 화면

파일 플러그인 사용 여부를 “Y”로 선택 시, Application-specific metric이 어떤 파일 포맷으로 수집되는지 선택 후 추가()버튼을 클릭한다. 현재 수집을 지원하는 파일 포맷은 binary, CSV, JSON이 있다.

그림 12-10 Agent 탭 화면

그림 12-27 서비스 상세 – Application 정보

그림 12-28 서비스 상세 – Agent 정보

서비스 형상

서비스 형상은 Provider, Cloud, Instance, Application, Service로 layer(층)를 나누어 각 layer에 해당하는 서비스 구성 요소의 정보를 출력한다. Layer에 출력된 구성 요소는 상태와 함께 출력되며 하나의 구성 요소를 선택 후 클릭 시 해당 구성 요소의 요약 정보가 출력된다.

PaaS

PaaS(Platform-as-a-service )는 해당 프로세스와 관련된 인프라 또는 플랫폼을 구축하고 유지 관리할 필요 없이 자체 애플리케이션을 개발, 실행 및 관리할 수 있도록 해준다. SPERO는 K8s를 기반으로 한 클라우드/클러스터와 연동을 하여, 클라우드/클러스터 시스템을 모니터링, 운영을 제공한다.

SPERO의 메인 메뉴에서 PaaS를 클릭하여 다음과 같은 기능을 조회할 수가 있다.

그림 14-1 PaaS 메뉴

클러스터 관리

클러스터(Cluster)는 컨테이너가 실행되는 인프라이다. 컨테이너는 어플리케이션의 배포 단위이자 실행 프로세스이다. 클러스터는 컨테이너 실행에 필요한 컴퓨팅 자원(CPU, Memory, 스토리지, 네트워크)을 제공한다.

클러스터는 노드(물리 머신 또는 가상 머신)로 구성된다. 하나 이상의 노드가 네트워크로 연결되어 있다. 클러스터는 분산 처리를 위해 고안된 아키텍처이다. 컨테이너를 클러스터에 배포하면 적절한 노드에서 실행된다. 이러한 과정을 스케줄링(Scheduling)이라 하는데, 쿠버네티스가 담당한다. 쿠버네티스는 클러스터에서 컨테이너 스케줄링과 관리를 담당한다.

클러스터는 노드의 추가를 통해 자원을 확장한다. 더 많은 자원이 필요하면 그만큼 노드를 추가 하고, K8s는 확장된 노드에 컨테이너를 배치하고 조정한다.

컨테이너 간 네트워크와 데이터 저장을 위한 스토리지도 클러스터의 구성 요소이다.

인그레스

인그레스는 외부 요청을 클러스터 내의 서비스로 라우팅 하기 위한 규칙을 정의하는 리소스이다. 클러스터 내의 다양한 서비스에 대한 접근을 중앙에서 관리하고, HTTP/HTTPS 트래픽을 적절한 서비스로 전달한다.

인그레스 목록을 선택하였을 때 목록을 조회하여 테이블 형식으로 출력한다. 인그레스 목록을 이름으로 검색하는 검색 기능 및 재조회 기능을 제공한다.
